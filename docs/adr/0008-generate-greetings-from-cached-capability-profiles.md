# Generate greetings from cached capability profiles

The job-seeker agent should derive a reusable Candidate Capability Profile from the candidate's resume, target intent, and stated requirements, then cache a non-sensitive summary until those inputs change. Full resume text may be used as transient input for capability-profile generation, but the cached profile, audit records, and per-job greeting-generation calls must not store or resend full resume text or other sensitive originals. Per-job LLM work should combine that cached profile with the target JD to generate a Personalized Greeting only after the final decision is `apply`; greeting generation does not grant Application Authorization. A Greeting Guard must validate the generated message before sending. When the final decision is `apply`, the system should prefer a guarded Personalized Greeting and fall back to the existing preset greeting selection when personalization is unavailable, generation fails, parsing fails, validation fails, or the LLM is unavailable. If no safe greeting text is available, starting chat may still proceed, but text sending must be skipped while image upload follows existing configuration. Long-term model configuration should distinguish `capability_profile_generation` from `greeting_generation`, while the MVP may reuse the currently enabled LLM configuration. CLI design should keep profile building and greeting generation as distinct capabilities that `run-once` can orchestrate after an apply decision; sending commands remain responsible for delivery rather than generation. Audit records should store greeting source, guard outcome, fallback reason, message summary, and character count, but not the full greeting text.

The MVP should be implemented in three slices: first capability-profile generation and caching, then Personalized Greeting generation plus Greeting Guard, then `run-once` integration after an apply decision. The existing preset greeting selection remains the delivery fallback throughout the migration.
