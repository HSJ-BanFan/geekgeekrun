# Relocate authorized jobs by jobId before browser actions

For `run-once --from-browser`, the action phase must use the authorized job's `jobId` as the Job Identity Anchor to relocate and verify the target job before performing real browser actions. Reusing the same browser/page can improve continuity, but it is an optional optimization rather than this decision's correctness boundary because the page may rerender, navigate, or otherwise drift while evaluation is running. Relocation is implemented inside the browser action layer without changing the CLI command shape. Relocation is bounded to the current job-source scope, such as the active query and city list context; it must not widen into global search or recall-keyword switching. If the current detail pane already has the authorized `jobId`, the action phase may treat that as successful relocation without clicking a job card. Otherwise, it scans the current rendered list and may scroll-load at most 5 times to find the authorized `jobId`; after clicking a matching job card, it must re-extract the detail pane and verify that its `jobId` still matches the authorized job before starting chat. Dry-run mode may report the planned `jobId` relocation and Job Identity Anchor, but must not click job cards or change the selected job. Skip decisions do not need relocation because no Application Authorization exists for a real chat action. If the authorized job has no `jobId`, or the action phase cannot relocate that `jobId`, or the post-click detail verification fails, confirmed browser actions must fail closed and only record audit evidence; in that failure state, `next_job` must also be skipped so the page cursor is not advanced from an unverified position. Relocation failures use stable reason codes: `JOB_IDENTITY_ANCHOR_MISSING`, `JOB_RELOCATION_NOT_FOUND`, `JOB_RELOCATION_DETAIL_MISMATCH`, and `JOB_RELOCATION_DETAIL_UNCONFIRMED`. Automated tests cover the browser action branch behavior with fake pages rather than a live BOSS session.
